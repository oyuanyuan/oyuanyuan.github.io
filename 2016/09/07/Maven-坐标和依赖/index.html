Maven 坐标和依赖

发表于 2016-09-07   |   分类于 Maven系列   |     |   阅读次数

坐标

Maven坐标元素包括：groupId，artifactId，version，packaging，classifier
groupId（必须）  定义当前Maven项目隶属的实际项目
artifactId（必须） 定义实际项目中一个Maven项目（模块）
version（必须） 定义Maven项目当前所处版本
packaging（可选） 定义Maven项目的打包方式，如 jar，war，pom，zip…，默认是jar
classifier（不能直接定义） 帮助定义构建输出的一些附属属性

依赖

简单的依赖配置（此配置在<project>下）：

groupId、artifactId、version 基本坐标
type 依赖类型，对应构件中定义的 packaging，可不声明，默认为 jar
scope 依赖范围，默认compile
optional 依赖是否可选
exclusions 排除传递依赖。

依赖范围

Maven在编译项目主代码、编译和执行测试代码、运行Maven项目会使用三种不同的classpath，分别是编译classpath、测试classpath、运行classpath。 

依赖范围就是用来控制依赖与这三种classpath的关系。

Maven的依赖范围有一下几种：

  1. compile：编译依赖范围，默认的依赖范围，对于编译、测试、运行classpath均有效
  2. test：测试依赖范围，只对测试classpath有效
  3. provided：已提供依赖范围，对于编译、测试classpath有效
  4. runtime：运行时依赖范围，对于测试、运行classpath有效
  5. system：系统依赖范围，对于编译、测试classpath有效，和provided一样，但是使用的时候必须使用systemPath元素显示的指定依赖文件的路径，由于此类依赖不是通过Maven仓库解析的，而且往往与本机系统绑定过，可能造成构建的不可移植，所以慎用
  6. import：导入依赖范围，不会对三种classpath产生影响
传递性依赖概念

Maven项目有spring-core依赖，spring-core又依赖commons-logging依赖，那么commons-logging依赖是Maven项目的一个传递依赖，Maven项目引依赖的时候，只需引入spring-core依赖，不需要考虑它还有什么依赖，Maven会自动引入传递性依赖

传递性依赖和依赖范围

依赖范围不仅可以控制依赖与三种classpath的关系，还对传递性依赖产生影响。
如果A依赖于B，B依赖于C，那么A对于B是第一直接依赖，B对于C是第二直接依赖
最左边一列表示第一直接依赖范围，最上面一行表示第二直接依赖范围，中间的交叉格表示传递性依赖范围

依赖范围影响传递性依赖表

compile test provided runtime
compile compile _ _ runtime
test test _ _ test
provided provided _ provided provided
runtime runtime _ _ runtime
依赖调节

当Maven项目中多个传递依赖对于同一个构件不同版本依赖时，Maven会遵循两个原则：

  1. 依赖路径不相同，短路径优先
  2. 依赖路径相同，先声明优先
可选依赖

当A依赖B，B依赖X或Y，X、Y就是可选依赖，可选依赖不会传递，B依赖中用true表明X，Y是可选依赖。A中需要显示声明具体使用X，还是Y。可选依赖不建议使用

排除依赖

A依赖B，B依赖C时，当B升级的时候，可能C正是不稳定的版本，所以我们不想使用C，想引入另外的稳定版本，排除掉了依赖传递，这就是排除依赖

归类依赖

一个项目里面，不同模块，但是依赖版本一样，可以把版本提出来定义成一个变量

优化依赖

查看已解析的依赖：查看Maven项目中所有的依赖
mvn dependency:list
查看依赖树：查看依赖之间的传递关系
mvn dependency:tree
查看使用但未声明的依赖与声明但未使用的依赖
mvn dependency:analyze
此命令只会分析编译主代码和测试代码需要用到的依赖，一些执行测试和运行时需要的依赖它就发现不了
